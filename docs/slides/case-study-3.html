Case Study 3

Azure Dual-Region Disaster Recovery

CloudInnovate — East US (Primary) | East US 2 (DR Standby)

Introduction to Azure — II  |  Hands-On Training

Scenario

CloudInnovate is a global SaaS company that needs a fully automated disaster recovery setup entirely within Azure, using two regions allowed by subscription policy.

Primary — East US

Serves all live traffic under normal conditions. Hosts the web tier, app tier, storage, and DNS zones.

DR Standby — East US 2

Identical infrastructure kept warm on standby. Activated automatically by Traffic Manager when East US becomes unhealthy.

Goals

  • RTO: < 60 seconds (DNS-based failover)
  • RPO: Near zero for stateless workloads
  • Automation: Fully declarative via Bicep / ARM templates

Key Concepts: Disaster Recovery

MetricDefinitionExample
RPORecovery Point Objective — max acceptable data loss"We can lose 15 min of data"
RTORecovery Time Objective — max acceptable downtime"Must be back within 1 hour"

DR Strategy Tiers

Tier 1: Hot Standby

Active-Active • RPO ~0s, RTO ~0s
Cost: Highest (double infra running)
Our lab uses Tier 2 (Warm)

Tier 2: Warm Standby

Active-Passive • RPO: minutes, RTO: minutes
Standby infra pre-deployed
Traffic Manager + pre-provisioned VMs

Tier 3: Cold Standby

Backup & Restore • RPO: hours, RTO: hours • Cost: Lowest
ARM templates + Azure Backup — deploy on demand

Key Concepts: Virtual Networks

VNet is the fundamental building block for private networking in Azure. It enables resources to communicate securely with each other, the internet, and on-premises networks.

Subnet Layout (per region)

VNet: 10.0.0.0/16  (65,536 addresses)
├── subnet-web     10.0.1.0/24   (251 usable) — Web tier VMs + LB
├── subnet-app     10.0.2.0/24   (251 usable) — Application tier VMs
├── subnet-db      10.0.3.0/24   (251 usable) — Database tier
└── subnet-gateway 10.0.4.0/27   (27 usable)  — VPN/ExpressRoute GW

Network Security Groups (NSGs)

SubnetRuleSourcePort
subnet-webAllow HTTPS/HTTPInternet443, 80
subnet-appAllow app traffic10.0.1.0/248080
subnet-dbAllow DB traffic10.0.2.0/245432

Key Concepts: Azure DNS

Public DNS Zone

cloudinn-app.example.com

  • Internet-facing domain resolution
  • A, CNAME, MX, TXT records
  • Hosted on Azure's global anycast network
  • NS records delegated from registrar

Private DNS Zone

internal.cloudinn.azure

  • Internal service discovery within VNet
  • Auto-registration of VM hostnames
  • Linked to primary VNet
  • Not reachable from the internet

DNS Delegation Flow

1
User queries cloudinn-app.example.com
2
Domain registrar NS records point to Azure DNS (ns1-XX.azure-dns.com)
3
Azure DNS resolves and returns the A record (Load Balancer IP)
4
Client connects directly to the Load Balancer

Key Concepts: Azure Traffic Manager

DNS-based global traffic load balancer. It does NOT proxy traffic — it tells clients which endpoint IP to use via DNS. Clients connect directly to the endpoint.

Priority Routing (Our DR Strategy)

Priority 1: East US (Primary)     ← All traffic here normally
Priority 2: East US 2 (DR)        ← Only if Priority 1 fails

Health Monitoring

SettingValue
ProtocolHTTPS
Path/health
Probe interval10 seconds
Tolerated failures2 consecutive
DNS TTL30 seconds

Worst-case failover time: ~20s detection + ~30s TTL = < 60 seconds

Key Concepts: Infrastructure as Code (Bicep)

Bicep is Azure's domain-specific language for deploying resources declaratively. It compiles to ARM JSON templates but is far more readable.

Why Bicep for DR?

  • Repeatability: Deploy identical infrastructure in both regions from the same template
  • Idempotent: Re-run safely — only changes what's different
  • Version controlled: Infrastructure changes tracked in Git
  • Modular: Separate templates for VNet, DNS, Infra, Traffic Manager

Failover Sequence

Normal State

User → DNS Query → Traffic Manager → Returns East US LB IP → User → East US

Failover Event (East US outage)

1
Traffic Manager probes East US endpoint every 10s via HTTPS /health
2
After 2 consecutive failures (~20s), endpoint marked Degraded
3
Traffic Manager returns East US 2 IP for all new DNS queries
4
TTL = 30s → propagates globally within ~30 seconds
5
New clients connect to East US 2 automatically

Failback (East US recovers)

1
Probes succeed again (2 consecutive) → East US reactivates as Priority 1
2
DNS returns East US IP • East US 2 returns to warm standby

Failover Test: CLI Commands

Linux / macOS (Azure CLI)

# Get Traffic Manager FQDN
TM_FQDN=$(az network traffic-manager profile show \
  --resource-group rg-cloudinn-primary-prod \
  --name tm-cloudinn-dr \
  --query dnsConfig.fqdn -o tsv)

# Before failover — should return East US IP
nslookup $TM_FQDN

# Simulate outage: disable East US endpoint
az network traffic-manager endpoint update \
  --resource-group rg-cloudinn-primary-prod \
  --profile-name tm-cloudinn-dr \
  --name endpoint-eastus-primary \
  --type azureEndpoints \
  --endpoint-status Disabled

# Wait ~60s then verify — should return East US 2 IP
sleep 60 && nslookup $TM_FQDN

# Re-enable (failback)
az network traffic-manager endpoint update \
  --resource-group rg-cloudinn-primary-prod \
  --profile-name tm-cloudinn-dr \
  --name endpoint-eastus-primary \
  --type azureEndpoints \
  --endpoint-status Enabled

Key Takeaways

Architecture

  • Active-Passive across two Azure regions
  • Identical infra in both regions (Bicep modules)
  • Traffic Manager = DNS-level failover
  • TTL 30s + 10s probes = ~50s max failover

Concepts Covered

  • RPO / RTO and DR strategy tiers
  • VNets, Subnets, NSGs
  • Public & Private Azure DNS
  • Azure Traffic Manager (Priority routing)
  • Infrastructure as Code with Bicep
